  • 索引设计是数据库性能优化的重要指标之一, 但同时索引机制无论从实现原道还是设计应用也非常复杂, 企业中往往由熟悉企业业务的数据库管理员(DBA)进行设计优化, 此处对于索引设计的学习主要以理解为主

1. 语法回顾

  • 概念
    • 为提高查询速度而提供的一种数据结构, 类似书的目录, 可以快速查询出数据, 而不是从头到尾的依次比对 (慢查询)
  • 优点
    • 提高查询的速度

  • 缺点

    • 增加数据库的存储空间
    • 减慢增删改速度(索引需要随之改变)

  • 设置索引

create table t_user(
id int not null,  
nickname varchar(20) not null,
index (nickname)  
)  # 建表时添加索引

# 修改表-添加索引
alter table t_user add index (nickname); 
# 修改表-删除索引 
alter table t_user drop index nickname;
  • 查看索引
show index from t_user;
  • 查看查询是否使用了索引
explain select * from t_user where nickname = 'zs';
  • 索引分类
    • 主键索引
      • 创建主键后自动生成
    • 二级索引
      • 唯一索引
        • 设置唯一约束后自动生成
      • 外键索引
        • 设置外键约束后自动生成
      • 自定义索引
        • 开发者手动创建的索引 index

 

2. 索引原理

  • Mysql索引采用的基本数据结构为 平衡树 (BTree), 其中 InnoDB引擎 又进一步升级, 拆分为 聚簇索引 和 非聚簇索引 两种

2.1 平衡树

  • Mysql的索引存储结构为B+tree结构, 是一种平衡多叉树
  • 利用 二分查找, 减少硬件检索次数, 提高查询效率

 

2.2 聚簇/聚集索引

  • 叶子节点上存放的是 数据行

  • InnoDB引擎的主键索引 使用这种存储方式

  • 示例表

create table t_user(
  id int not null primary key auto_increment,  
  name varchar(10) not null,
  birthday datetime,
  index (name)  // 设置二级索引
)
id name birthday
1 Tom 1996-01-01
2 Jann 1996-01-02
3 Ray 1996-01-08
4 Lily 1996-01-13
5 Steve 1996-01-15

 

2.3 非聚簇索引/非聚集索引

  • 叶子节点上 不包含行记录的所有数据, 只包含索引数据 和 一个"书签", 该书签用于查询到索引对应的行数据
  • InnoDB的二级索引 和 MyISAM的所有索引 使用这种存储方式

 

  • InnoDB的二次查询
    • InnoDB二级索引的叶子节点上存放的"书签" 是 数据行的主键, 所以二级索引需要进行 二次查询

 

  • MyISAM的索引查询
    • MyISAM引擎的所有索引都采用非聚簇索引, 且 索引和数据行存储在不同的文件中, 其"书签"存储的是 数据行的物理地址 (磁盘位置)

  • 索引查询

InnoDB引擎默认将所有表的索引和数据都保存在一个文件中, 称为共享表空间, 默认名为 ibdata1, 也可以通过设置让每个表对应一个文件 MyISAM引擎每张表的数据分别放在两个文件中, 数据行在 表名.myd 中, 而索引在 表名.myi 中

  • 总结:
    • 主键一般选择趋势递增, 以便让数据顺序添加到B+Tree中, 避免因移动数据而影响插入速度
    • InnoDB的主键查询快
    • MyISAM的所有索引查询速度相同

 

3. 索引机制

3.1 联合索引

  • mysql支持多列创建联合索引
create table t_user(
  id int not null primary key auto_increment,  
  name varchar(10) not null,
  age tinyint default 0,
  birthday datetime,
  index k_name_age (name, age)  # 设置联合索引
)
  • 注意点:
    • 联合索引使用最左前缀原则
    • 如: where mobile /where mobile and age 可以使用索引, 但 where age 不能使用索引

3.2 覆盖索引

  • 如果一个索引包含(或者说覆盖)所有需要查询的字段的值, 称为 覆盖索引
  • 如果查询触发了覆盖索引, 将 直接从索引中提取出数据, 而不会再去查询数据行(二次查询)
  • 合理利用覆盖索引机制, 可以优化二次查询, 提高查询效率

 

4. 索引优化

适合建立索引的情况

  • 经常条件过滤的字段 Where
  • 连接查询的连接字段 on
  • 经常排序的字段 order by

不适合建立索引的情况

  • 更新非常频繁的字段不合适建立索引 (更新频率>查询频率)
  • 选择性低的字段, 不适合单独建立索引(索引效果差)

索引失效的情况

  • 违背最左前缀原则 --- 无法使用索引
  • 跳过索引中的列 --- 只使用部分索引
  • 某列属于联合索引, 对该列进行范围查询 --- 只使用部分索引, 联合索引中该列右边的列无法使用索引
  • !=, is null, is not null --- 无法使用索引
  • like以通配符开头 --- 无法使用索引
  • 用or连接条件 --- 无法使用索引

索引优化

  • 语句
    • where和on的字段设计索引
    • order by / group by字段 尽量使用索引
  • 效率
    • 不要建立太多的索引 (查询一般只会用一个索引)
    • 尽量让索引在多个接口中可用
    • 尽量将索引建立在选择性高的字段上
    • 尽量使用覆盖索引, 减少select *
  • 联合索引
    • 单键索引的过滤率在90%以下 且 高并发场景 , 优先建立联合索引
    • 联合索引尽量全值匹配
    • 注意联合索引的设计顺序, 选择性高的放最左, 范围查询放最后
  • 失效
    • 索引列不要为可空列
    • 尽量少用like 尽量少用!= 尽量少用or
