1. 数据库引擎

  • 总的来说，MySQL 可以看成是二层架构:
    • 第一层: SQL Layer，在 MySQL 数据库系统处理底层数据之前的所有工作都是在这一层完成的，包括网络交互，权限判断，SQL解析，SQL优化，缓存处理等;
    • 第二层: Storage Engine Layer, 即存储引擎层，负责底层数据存取操作实现部分，由多种存储引擎共同组成

  • 数据库引擎代表着数据存取的不同解决方案
  • 每张表可以选择不同的数据库引擎
  • 设置格式:
CREATE TABLE 表名 (
  字段1;
  字段2;
) ENGINE=MyISAM;  # ENGINE=数据库引擎名

 

  • 下面主要介绍和对比常用的两种: InnoDB 和 MyISAM

InnoDB

  • mysql5.5开始, 成为建表默认采用的数据库引擎
  • 特点:
    • 支持事务 (回滚/提交/ACID特性/多版本并发控制等)
      • 数据恢复可使用事务日志(undo/redo log), 恢复速度快
    • 支持行级锁 & 表级锁
      • 并发访问时效率高
    • 支持外键约束
      • 为数据提供完整性保护
    • 插入/更新/主键查询快
    • 需要内存和硬盘多
  • 使用场景:
    • 常规推荐使用

MyISAM

  • mysql5.5之前, 是建表默认采用的数据库引擎
  • 特点:
    • 不支持事务
    • 只支持表级锁
    • 不支持外键约束
    • 批量插入 / 查询 / 聚合查询 速度快
  • 使用场景:
    • 简单, 适合小型项目 / 只读系统 (如: 内部管理系统)

项目应用

  • 项目中多数表还是采用默认的InnoDB引擎
  • 系统公告表选择了 MyISAM
    • 因为该表数据基本不会修改, 不存在大量并发写操作, 也就不需要行级锁提供并发能力
    • 和其他表没有关联, 不需要事务提供原子性等支持
    • 公告数据查询多, MyISAM简单, 查询会更快

 

2. 字符集

  • 建表默认的字符集为 utf-8
  • 但使用 utf-8 保存数据时, 如果数据中包含表情符号会崩溃
    • 原因是 utf-8编码 最大字符长度为3字节, 而 unicode中 大编码实现的表情符号(emoji) 为4字节
  • 如果想要让表中的字段支持表情符号, 需要设置编码方式为 utf8mb4
  • 设置建表采用的字符集:
CREATE TABLE 表名 (
  字段1;
  字段2;
) ENGINE=MyISAM DEFAULT CHARSET=utf8mb4;  # CHARSET=字符集

 

3. 注释

  • Mysql可以通过 COMMENT参数 给 字段和表 设置 说明注释
  • 该注释在使用 show create table xx; / show full columns from test;语句时 可以显示出来
  • 设置格式:
CREATE TABLE 表名 (
  字段1 类型 COMMENT '字段1注释',
) ENGINE=InnoDB DEFAULT CHARSET=utf8 COMMENT='表注释';
